# Audit Trail Enhancement Design

## Context

CEMS-MY is a Laravel-based Currency Exchange Management System for Malaysian Money Services Businesses, compliant with Bank Negara Malaysia (BNM) AML/CFT requirements. The system requires comprehensive audit logging for regulatory compliance, security monitoring, and operational visibility.

The existing audit trail captures several event types but has significant gaps across MFA, stock transfers, journal workflow, compliance alerts/cases, EDD templates, regulatory reports, data breach events, and automated compliance monitoring.

## Goals

1. **BNM/Regulatory Compliance** — Ensure all required events are captured for AML/CTF audits
2. **Security Hardening** — Track suspicious patterns, failed authorizations, data access
3. **Operational Visibility** — Understand user behavior, debug issues, track SLA
4. **Tamper Evidence** — Maintain hash-chained audit log for forensic integrity

## Design

### Architecture

Extend `App\Services\AuditService` with domain-specific logging methods. Each method follows the established pattern:
- Rich context capture (who, what, when, where)
- Standardized action names using `snake_case`
- Severity levels appropriate to the event type
- Old/new values for state changes
- Tamper-evident hash chaining

### Action Naming Convention

Format: `{resource}_{action}` in snake_case
- Example: `mfa_verification_failed`, `stock_transfer_dispatched`, `journal_entry_approved`

### Severity Mapping

| Severity | Usage |
|----------|-------|
| `INFO` | Successful logins, MFA verify, normal CRUD operations |
| `WARNING` | Permission denied, failed verifications, compliance alerts created |
| `ERROR` | Sanction hits, structuring detected, velocity breaches, AML rule triggers |
| `CRITICAL` | STR submitted to goAML, data breach detected, bulk sanctions hits |

### Data Captured Per Event

All events capture:
- `user_id` — Who performed the action
- `ip_address` — Client IP
- `user_agent` — Browser/client info
- `session_id` — Session identifier
- `entity_type` — Resource type (e.g., "Transaction", "Customer")
- `entity_id` — Resource ID
- `old_values` — Previous state (for changes)
- `new_values` — New state
- `severity` — INFO/WARNING/ERROR/CRITICAL
- `timestamp` — ISO 8601 format
- `previous_hash` — Chain link to prior entry
- `entry_hash` — SHA-256 hash of this entry

### Implementation Notes

1. **Backward Compatibility** — Existing logging calls (`SystemLog::create` in controllers) should be migrated to use the new service methods over time, but can coexist during transition

2. **Background Jobs** — AML monitoring events fired from queues should use `logAmlMonitorEvent` with the job's queue context

3. **Sensitive Data** — Password values, recovery codes, and secrets must never be logged in `old_values` or `new_values`

4. **Rate Limiting** — Permission denied events should be rate-limited to avoid log flooding from brute-force attempts

## Files to Modify

1. `app/Services/AuditService.php` — Add 18 new logging methods
2. Controllers to add logging calls (see implementation plan)
3. `tests/Unit/AuditServiceTest.php` — Add tests for new methods

## Out of Scope

- Modifying existing hash chain algorithm (already tamper-evident)
- Changes to `SystemLog` model structure
- UI changes for audit log viewer
- Log retention/rotation policies (handled by `LogRotationService`)
